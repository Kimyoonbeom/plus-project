
# 📚 booklog – 독서의 모든 순간을 기록하다

**booklog**는 사용자의 독서 활동을 기록하고, 책을 기반으로 다양한 이벤트 및 쿠폰 기능을 제공하는 **이커머스 + 커뮤니티 기능**이 결합된 Spring Boot 기반 웹 프로젝트입니다.

---

## 🎯 프로젝트 핵심 목표

### 🧑‍💻 윤태님
- **개발 속도 향상**: 코드 변경 사항을 빠르게 테스트하고 배포하여 개발 주기 단축
- **배포 자동화**: 수동 개입 없이 신뢰성 있는 자동 배포 구현
- **코드 품질 향상**: 지속적인 테스트로 버그 조기 발견 및 수정
- **협업 효율성 증가**: 병합과 테스트 병행 가능
- **안정적인 운영**: 환경 차이로 인한 문제 최소화
- **빠른 피드백 루프**: 코드 푸시 시 테스트 및 알림 즉시 수행

### 🧑‍💻 소미님
- **쿼리 성능 최적화**: 조회 빈도 높은 컬럼에 인덱싱 적용
- **JPA 인덱스 설정**: `@Table(indexes = {...})` 활용
- **불필요한 Full Scan 방지**

### 🧑‍💻 선영님
- **JWT 기반 인증**: 세션 일관성과 API 접근 제어
- **캐싱 기능을 적용하여 검색 응답 속도 75% 개선**: 1,884ms → 474ms

### 🧑‍💻 재호님
- **재고 동시성 제어**: Redisson 활용
  - 쿠폰 재고: Redisson 사용, ‘선착순’ 순서 보장 필수
  - 책 재고: 트래픽 적지만 ‘선착순’ 순서 중요 → Redisson 선택

### 🧑‍💻 윤범님
- **OAuth2 도입**: 인증 절차 간소화, 보안성, 확장성 강화.

  - OAuth2 표준 프로토콜을 도입하여 비밀번호 저장 없이 안전하게 사용자 인증을 처리
  - JWT 기반 토큰 인증과 결합하여 API 접근 제어와 세션 관리의 일관성 확보

  - 액세스 토큰 기반 인증으로 각 API 요청에 대한 권한 검증을 강화.
  - 민감 정보(비밀번호 등) 저장/전송 최소화로 개인정보 유출 위험 감소.

## 🧵 KEY SUMMARY

### 🍁 성능 개선: Redis 및 로컬 캐시 도입 효과

**📌 테스트 환경 요약**
- 사용자 수(Vuser): 1명씩 초당 증가, 최대 500명
- 테스트 시간: 1분
- 시나리오: 검색 API 응답 캐싱 전략별 성능 비교(DB vs 로컬 캐시 vs Redis)

| 항목              | DB           | 로컬 캐시       | Redis          |
|------------------|--------------|------------------|----------------|
| **TPS**          | 111.0        | 480.2            | 457.8          |
| **Peak TPS**     | 191.5        | 936.5            | 1,115.5        |
| **Mean Time (ms)**| 1,884.54     | 437.75           | 474.33         |
| **Executed**     | 4,175        | 23,975           | 25,544         |
| **Successful**   | 4,175        | 19,578           | 18,586         |
| **Errors**       | 0            | 4,397            | 6,958          |

**📈 인사이트**
- **평균 응답 속도**: DB 대비 Redis는 약 **75% 이상 속도 개선** (1,884ms → 474ms)
- **TPS (초당 처리량)**: Redis는 DB 대비 **4배 이상 높은 TPS**
- **최대 TPS (Peak TPS)**: Redis가 가장 높은 최대 처리량 기록 (**1,115 TPS**)
- **실패율 증가 요인**: TPS 및 트래픽 증가에 따른 Redis 오류는 TTL 만료, 캐시 미스 등의 원인 가능성 존재

**📌 결론**
- **Redis 캐싱 적용 시**: 대규모 트래픽 상황에서 처리량과 응답 속도 모두 크게 개선됨
- **추가 고려사항**: 오류율 최소화를 위해 TTL 조정, fallback 로직 등 안정성 보완 필요

### 🍁 **트러블 슈팅 : EC2 인스턴스 SSH 22번 포트 연결 오류**

#### 1. 문제 발생
- CD 과정 중 EC2 인스턴스에 접속 시 다음과 같은 에러 발생:

  ```
  ssh: connect to host x.x.x.x port 22: Connection timed out
  ```

  또는

  ```
  kex_exchange_identification: read: Software caused connection abort
  ```

---

#### 2. 해결 방안

##### 🔐 1. 보안 그룹(Security Group) 확인
- SSH (TCP/22) 규칙이 추가되어 있는지 확인
- 소스 IP 설정:
  - `0.0.0.0/0`: 모든 외부 접근 허용 (테스트용, 보안 취약)
  - `본인 공인 IP/32`: 제한된 접근 (권장)
- IPv4 기준인지 확인 (`::/0`는 IPv6)

> 콘솔 경로: EC2 > 인스턴스 > 보안 > 보안 그룹 > 인바운드 규칙

---

##### 🌐 2. 네트워크 ACL(Network ACL) 확인
- 서브넷에 연결된 네트워크 ACL에서 포트 22 허용 여부 확인
  - 인바운드: TCP 22 허용
  - 아웃바운드: Ephemeral 포트(1024~65535) 허용

---

##### 🛣️ 3. 라우팅 테이블 확인
- VPC 라우팅 테이블에 다음 경로 존재 여부 확인:
  - `0.0.0.0/0 → 인터넷 게이트웨이(igw-xxxx)`

---

##### 🌐 4. 인터넷 게이트웨이 연결
- VPC에 인터넷 게이트웨이가 연결되어 있어야 함
- 해당 서브넷의 라우팅 테이블에 IGW가 연결돼 있어야 함

---

##### 🌍 5. 퍼블릭 IP 확인
- EC2 인스턴스에 퍼블릭 IPv4 주소가 할당되어 있어야 함
- 자동 할당 또는 탄력적 IP(EIP) 설정 가능

---

##### 👤 6. OS 사용자 이름 확인
- 배포 OS에 따른 사용자 이름 확인:
  - Ubuntu: `ubuntu`
  - Amazon Linux: `ec2-user`
  - RHEL/CentOS: `ec2-user`, `centos`
  - Debian: `admin`, `debian`
- 잘못된 사용자 이름 → 연결은 되지만 로그인 실패

---

##### 🔑 7. 키 페어 (.pem) 확인
- 접속 시: `ssh -i ~/경로/Book.pem ...`
- 퍼미션 오류 없는지 확인 (`chmod 400 Book.pem`)
- 인스턴스 생성 시 등록된 키 페어와 일치하는지 확인

---

##### 🛡️ 8. VPN, 사내망, 방화벽 확인
- VPN/사내망에서 포트 22 차단 가능
  - ➜ 모바일 핫스팟으로 우회 테스트 가능

---

##### 📊 9. 인스턴스 상태 확인
- 인스턴스 상태: **"실행 중"**
- 상태 검사: 2개 모두 통과 여부 확인

---

##### 📡 10. ping 실패 여부 확인
- ping: ICMP 프로토콜 / SSH: TCP 22포트 사용
- ping도 안 될 경우 → NACL/라우팅 문제
- ping 가능하지만 SSH 실패 → 보안 그룹 설정 문제일 가능성 높음

## ⚙ 기술적 고도화: 동시성 제어 전략


| 구분 | 방식 | 응답시간 |
|------|------|----------|
| 책 재고 | 낙관적 락 | 1,463ms |
| 책 재고 (재시도) | 낙관적 락 + 재시도 | 15,765ms |
| 책 재고 | Redisson | **3,642ms** |
| 쿠폰 재고 | 비관적 락 | 702ms |
| 쿠폰 재고 | Redisson | **3,858ms** |

### 선택 이유
- Redisson(fairLock) 사용으로 **순서 보장 + 성능 향상**
- 낙관적 락 대비 **4배 빠름**, 재시도 불필요
- 비관적 락보다 느리지만 **순차 처리 필수 시 Redisson 우위**

#### 낙관적 락과 비관적 락의 선택지


- **비관적 락**  
  비관적 락으로 데이터를 조회하면 해당 트랜잭션이 끝나기 전까지는 데이터에 대한 Insert 작업이 불가능합니다.
  - 단점
    - 트래픽이 많은 경우 성능 저하 발생 및 타임아웃 문제가 발생합니다
    - 타임아웃을 잘 걸지 않으면 데드락 발생 위험이 생깁니다
    - 확장성에 취약합니다

- **낙관적 락**  
  낙관적 락은 충돌 발생 시 롤백 처리를 요구하며, 충돌 비용이 높습니다.
  - 단점
    - CPU 점유율이 상승하고, 예상치 못한 오류 발생 가능합니다
    - 충돌 시 재시도 로직이 필요합니다
    - 순서가 보장되지 않습니다
    - 트래픽이 심한 경우 성능이 급락합니다

- **Redisson**  
  Redisson은 Redis의 싱글 쓰레드 방식을 이용한 분산 락 방식이며 그 중 fairLock을 사용하면 순서까지 보장이 가능한 동시성 제어 방식입니다.
  - 단점
    - Redis에 의존하기 때문에 Redis가 다운되면 락 제어도 불가능해집니다
    - 모든 과정이 끝나도 유지시간 동안 Lock을 유지하고 있기 때문에 UnLock 해주는 과정이 필요합니다
    - Key로 관리하기 때문에 Key가 충돌하지 않게 주의를 요구합니다


책 재고의 경우 Redisson 방식을 적용하기 전에는 낙관적 락 방식을 적용하였습니다. 책의 경우 트래픽이 몰리는 빈도가 낮기 때문에 비관적 락 보다는 낙관적 락 방식이 더 맞는 방식이라고 생각하였기 때문입니다.
그러나 낙관적 락을 적용한 후 조금 더 공부해보는 과정에서 '재고 관리(100개의 재고에 100명이 접근할 경우 재고는 0이 되어야 한다)' 측면과 '순서 관리(100개의 재고에 1000명이 접근시도할 경우 먼저 요청한 순서대로 응답해야 한다)' 라는 측면에서 Redisson 방식이 떠올랐고 실제로 테스트 코드를 돌려본 결과 Redisson 방식이 재시도 로직을 포함한 낙관적 락 방식보다 4배 가량 빠른 결과가 나와 Redisson 방식을 채택하게 되었습니다.

쿠폰 재고의 경우 Redisson 방식을 적용하기 전에는 비관적 락 방식을 적용하였습니다. 쿠폰의 경우 트래픽이 몰리는 경우가 빈번하고 정합성이 매우 중요하다고 생각하였기 때문입니다.
그러나 비관적 락을 적용한 후 생각해보니 비관적 락은 순서를 보장하지 않는다는 단점이 있었고 이를 보완하기 위해 순서를 보장해줄 수 있는 Redisson(fairLock) 방식을 채택하게 되었습니다.

---

### 적용 후

- **CPU 점유율:** 낙관적 락 적용 76% → Redisson 방식 적용 15% 으로 약 60%point 향상
- **TPS:** 낙관적 락 적용 63 → Redisson 방식 적용 276으로 약 200 이상 향상
